## Key Findings (from initial exploration 2026-07-13)

### Architecture
- `src/` — daemon (`digiasset_core`). RPC server at `src/RPC/Server.cpp`;
  methods in `src/RPC/Methods/*.cpp`, one file per method, each with a
  matching `.html` doc file. `src/RPC/MethodList.cpp` is **auto-generated by
  CMakeLists.txt** — just add a new `<name>.cpp` in Methods/ and it gets registered.
  Unknown methods are forwarded to the DigiByte Core wallet (see Server/DigiByteCore).
- `cli/` — `digiasset_core-cli` is a **generic JSON-RPC pass-through**: any new RPC
  method is automatically available from the CLI. So "CLI support" == "RPC method exists".
- `qt/` — GUI app. Launches daemon, shows splash w/ sync progress, then a QTabWidget
  with only one tab (`tabs/SyncTab`). New tabs go in `qt/tabs/`, use `RPCLoader` to talk
  to the daemon. Add new tabs in `qt/main.cpp` (`updateLoadingProgress`, sync==0 branch).
- `web/` — `digiasset_core-web` = documentation server (port 8090) serving `web/index.html`
  + `web/rpc/*.html` (DigiByte wallet RPC docs) and custom method docs from
  `src/RPC/Methods/*.html`.

### Task 2 & 3 — what exists / what doesn't
- `send`, `sendtoaddress`, `sendmany` RPCs: **DGB only** — they just translate
  DigiByte domains → addresses and forward to the wallet. NOT asset-aware.
- `getaddressholdings <address>` RPC: returns {assetIndex: qty} for ONE address from the
  local DB. No wallet-wide balance, no asset names/decimals applied.
- `DigiByteTransaction::addDigiByteOutput()` (src/DigiByteTransaction.cpp:840) is a
  **stub (todo comments only)**. `addDigiAssetOutput` is declared in the header but NOT
  implemented. Building new asset transactions (issuance and transfer encoding) must be
  written. Decoding exists (`decodeAssetTX`, `decodeAssetTransfer`) — use it as the spec
  reference, plus the DigiAsset v3 protocol.
- KYC/asset UTXO data lives in the local sqlite DB (`Database::getAddressHoldings`, etc.).
  Wallet keys/signing live in DigiByte Core — plan: build raw tx locally, then
  `fundrawtransaction`/`signrawtransactionwithwallet`/`sendrawtransaction` via the wallet.
  CAREFUL: fundrawtransaction must not select asset-bearing UTXOs as fee inputs —
  need to lock or explicitly select asset UTXOs (`lockunspent` or explicit inputs).
